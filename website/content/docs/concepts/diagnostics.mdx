media is wired into the same four "colas" (glue points) every `@dudousxd/nestjs-*` library shares. None of them are required to use media — they're how it *composes* with the rest of the ecosystem when you want it to.

## 1. Diagnostics channels

`MediaLibrary` publishes to `node:diagnostics_channel` with the standard ecosystem envelope. This is the lowest-coupling integration there is: media doesn't import Telescope or know it exists — it just emits, and anything that cares subscribes. When nobody's listening, publishing is a no-op (a cheap `hasSubscribers` check), so it's always on by default with no measurable cost.

```ts
// channel: nestjs:media:<event>   where event ∈ attach | delete | conversion
{ ts: number, lib: 'media', event: 'attach', payload: { id, ownerType, ownerId, collection, disk, path, size, mimeType } }
```

Subscribe to it yourself if you want custom metrics or audit logging:

```ts
import { subscribe } from 'node:diagnostics_channel';

subscribe('nestjs:media:attach', (msg) => {
  metrics.increment('media.attach', { disk: (msg as any).payload.disk });
});
```

Turn emission off with `emitDiagnostics: false` on the `MediaLibrary` options if you never want it. See [Diagnostics](/docs/diagnostics) for the cross-ecosystem convention this follows.

## 2. Telescope

[`@dudousxd/nestjs-media-telescope`](/docs/packages/telescope) ships a `MediaWatcher` that subscribes to those channels and records a `media` entry per event — so attaches, deletes, and conversions show up in the [Telescope](/docs/telescope) console next to your requests, queries, jobs, and mails, correlated under the same batch.

```ts
import { MediaWatcher } from '@dudousxd/nestjs-media-telescope';
// add new MediaWatcher() to your telescope watcher list
```

That's the entire integration: media emits, the watcher records. No changes to your media code, no coupling between the packages beyond the channel name.

## 3. Codegen

[`@dudousxd/nestjs-media-codegen`](/docs/packages/codegen) contributes a typed media client to your generated output, so the front-end gets `uploadMedia()` + `mediaUrl()` with no hand-written upload protocol. The emitted file is a *thin binding* over the runtime client (it bakes in your tus base path and re-exports), which is the point of the next section. See [Codegen](/docs/codegen).

## 4. React

[`@dudousxd/nestjs-media-react`](/docs/packages/react) provides `useMediaUpload()` and `<MediaUploader/>`, built on the same resumable client — progress, status, and resume handled for you.

<Callout type="info" title="One protocol, many surfaces">
The tus client lives exactly once, in `@dudousxd/nestjs-media-client`. The React hook wraps it; the codegen extension re-exports it. There is a single implementation of the wire protocol, so a protocol change is a one-file change — the front-end surfaces follow automatically.
</Callout>

## Why this shape

Every glue point here is *opt-in and decoupled*. media is useful with none of them (it's a storage + media library on its own), better with each of them, and never forced to depend on Telescope, the codegen runtime, or React to function. That's the ecosystem contract: libraries emit on shared channels and ship satellite packages, so you assemble exactly the surface you need.
